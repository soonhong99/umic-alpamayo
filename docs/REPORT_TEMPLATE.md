# 조사/변경 보고서 작성 표준

`docs/` 아래의 날짜별 조사·변경 보고서(`YYMMDD_제목.md`)는 아래 형식을 따른다. 국내 공공·기업
문서에서 쓰는 개조식·항목번호 체계([행정업무의 효율적 운영에 관한 규정 시행규칙](https://www.law.go.kr/LSW/lsInfoP.do?lsiSeq=119635)
기준)를 이 repo의 코드 중심 markdown 문서에 맞게 적용한 것이다.

## 1. 문서 header

문서 맨 위에 이 표를 둔다.

```markdown
| 항목 | 내용 |
|---|---|
| 문서명 | (제목) |
| 작성일 | YYYY-MM-DD |
| 작성자 | (이름 또는 계정) |
| 버전 | (CHANGELOG.md의 개정이력 요약 표와 같은 버전 번호) |
| 관련 커밋 | (있으면 커밋 해시 링크) |
```

## 2. 표준 목차

아래 다섯 절을 이 순서로 둔다. 절 이름은 그대로 쓰지 않아도 되지만(예: "배경 및 목적" 대신
"왜 이 조사를 했는가"), **역할은 유지**한다.

1. **결론 요약**: 한두 문단, 무엇이 얼마나 좋아졌는지·정확도는 어떤지를 숫자로 먼저 제시.
2. **배경 및 목적**: 왜 이 조사/변경이 필요했는지.
3. **현황 및 발견사항**: 무엇을 찾았는지, 어떻게 확인했는지(재현 가능한 스크립트/명령 포함).
4. **조치 내용 및 검증 결과**: 무엇을 어떻게 고쳤고, 정확도·성능을 어떻게 검증했는지.
5. **결론 및 향후 계획**: 남은 일, 한계, 다음 조사 대상.

## 3. 항목 번호 체계

markdown 제목(`##`, `###`)을 최상위 구분자로 쓰고, 그 안의 나열은 국문 공문서 순서
(1. → 가. → 1) → 가) → (1) → (가))를 따른다. 표나 코드 블록이 섞인 기술 문서 특성상
`##`/`###` 두 단계까지는 markdown 제목을 쓰고, 그 아래 세부 나열부터 국문 순서를 적용하는
것을 권장한다(강제 아님. 표나 목록으로 충분히 명확하면 그대로 둔다).

## 4. 서술 방식

- **개조식**: 문장을 명사형으로 끝맺거나 짧게 끊는다("~함", "~됨", "~확인").
  단, 주어와 대상이 불분명해지지 않도록 주의한다(개조식의 가장 흔한 실패 지점).
- **탑다운**: 결론을 항상 먼저 쓰고, 근거·과정은 그 다음에 온다.
- **측정값과 개념을 구분**: "실측했다"와 "이론상 그럴 것이다"를 문장에서 명확히 나눈다.
- **em dash(—) 미사용**: 콜론(:), 괄호, 쉼표로 대체한다.

## 5. 문서 하단 개정이력 표

문서 자체가 나중에 수정되면(오류 정정, 후속 조사 반영 등) 아래 표를 문서 맨 끝에 둔다.
`CHANGELOG.md`의 저장소 전체 이력과는 별개로, **이 문서 하나의** 수정 이력이다.

```markdown
## 개정이력

| 버전 | 개정일자 | 개정내용 | 작성자 |
|---|---|---|---|
| 1.0 | YYYY-MM-DD | 최초 작성 | (이름) |
```
